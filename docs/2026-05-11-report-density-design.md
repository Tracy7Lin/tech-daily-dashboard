# 科技日报信息密度增强设计

## 目标

在不重做采集、分类与主题聚合的前提下，同时提升：

- 首页可读性
- 日报详情页的信息密度

本轮优先复用已有 `DailyReport -> TopicCluster -> CompanyReport -> EnrichedEntry -> RawEntry` 数据结构，只在渲染层做派生展示，必要时补少量格式化辅助函数，不新增新的持久化数据模型。

## 现状问题

当前 JSON 产物已经包含较多结构化信息，但 HTML 页面只展示了：

- 一条中文摘要
- 标签与重要度
- 原文链接

结果是页面读起来偏“压缩面板”，而不是“日报”：

- 首页主题卡没有把主题内的代表事件露出来
- 首页缺少跨公司重点观察入口
- 公司卡没有展示来源、发布时间、分类、对比角度
- 原始摘要没有被利用，导致信息量损失
- 日报详情页没有独立的“重点观察”层，阅读顺序不够明确

## 设计原则

- 高内聚低耦合：不把展示需求反向塞回分类器或流水线
- 优先复用：尽量从现有 `report.json` 派生，不新造状态
- 信息增量可控：增加信息量，但不把页面做成原始 JSON 展开器
- 首页与详情分工明确：首页偏导航与判断，详情偏展开与查阅

## 方案

### 1. 首页增强

首页新增 `重点观察` 区块，位于 hero 之后、主题对比之前。

每条重点观察基于全局高重要度条目派生，展示：

- 公司名
- 标题
- 中文摘要
- 原始摘要提炼后的补充说明
- 标签 / 分类 / 来源 / 发布时间
- 原文链接

首页主题卡扩展展示：

- 主题涉及公司数
- 主题内条目数
- 主题摘要
- 差异对比
- 趋势判断
- 主题内 2-3 条代表事件，每条带公司、标题、中文摘要

### 2. 日报详情页增强

详情页新增与首页一致但更完整的 `重点观察` 区块，优先展示 5-8 条。

公司动态卡扩展为“摘要 + 元信息 + 原文补充”结构：

- 中文摘要
- 来源 / 发布时间 / 分类 / 重要度 / 对比角度 / 标签
- 原始摘要清洗后的补充说明
- 原文链接

这样详情页既保留速览，也允许用户从原始摘要补充上下文，而不是必须跳出站点。

### 3. 主题与公司卡的信息边界

本轮不新增新的主题算法，只优化展示表达。

渲染层负责：

- 选择代表事件
- 格式化发布时间
- 清洗原始摘要 HTML
- 截断过长补充文本

流水线仍只负责生成结构化 `DailyReport`。

## 影响文件

- `src/tech_daily/render.py`
  - 增加重点观察派生逻辑
  - 增加原始摘要清洗与截断逻辑
  - 扩展主题卡与公司卡渲染
- `templates/index.html`
  - 新增重点观察区块
- `templates/daily.html`
  - 新增重点观察区块
  - 提升详情层信息排布
- `tests/test_render.py`
  - 补充首页与详情页渲染测试

## 非目标

- 本轮不改抓取器
- 本轮不改主题分类算法
- 本轮不引入 LLM 二次总结
- 本轮不做前端框架化改造

## 验收标准

- 首页出现独立的 `重点观察` 区块
- 首页主题卡能看到主题内代表事件，而非只有链接列表
- 公司卡能看到来源、发布时间、分类、对比角度、标签
- 详情页信息量明显高于首页，且不需要依赖跳转原文才能理解上下文
- 现有测试保持通过，并新增针对渲染增强的回归测试
