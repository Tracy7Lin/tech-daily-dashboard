# 科技日报专题跟踪 Agent 设计文档

## 1. 文档目的

这份文档用于定义 `tech-daily-dashboard` 的第三版 agent 能力：`专题跟踪 agent`。

当前系统已经具备：

- v1：当日情报判断
- v2：跨日追踪

第三版的目标不是扩大抓取范围，也不是引入外部搜索，而是在现有三层结果之上增加一个更聚焦的“专题观察”能力：

- 识别最近几天最值得跟踪的主题候选
- 选出一个主专题
- 围绕这个主专题输出更像分析文章的持续跟踪结果
- 同时输出 Markdown 报告与页面中的 `专题跟踪` 区块

## 2. 背景与动机

当前日报已经能回答两类问题：

1. `今天发生了什么`
- 由主日报和 v1 当日情报判断回答

2. `最近几天哪些主题和公司在变化`
- 由 v2 跨日追踪回答

但仍然缺少第三类更有解释力的问题：

- 在最近几天的候选主题里，哪一个最值得继续盯
- 这个主题为什么值得继续关注
- 哪些公司在持续参与
- 这些公司在同一主题下各自的切入点有什么差异
- 明天是否还值得把注意力继续放在这个专题上

因此，第三版 agent 的目标是：
**从最近几天的日报结果中挑出一个主专题，并生成一份可阅读、可归档、可在页面中快速消费的专题跟踪结果。**

## 3. 设计目标

本轮设计目标如下：

1. 从最近 N 天日报中识别 2-3 个候选主题
2. 选出 1 个主专题
3. 生成围绕该专题的持续跟踪结果
4. 明确：
- 该专题为什么值得看
- 哪些公司持续参与
- 各公司的切入点差异
- 主题最近几天是升温、扩展、分化还是回落
- 明天是否值得继续跟踪
5. 同时产出：
- Markdown 报告
- 页面中的 `专题跟踪` 区块
6. 保持失败可降级：专题跟踪失败时，不影响主日报、v1 与 v2
7. 在必要层使用现有 agent/LLM 能力，但不扩大到自主搜索与外部研究

## 4. 非目标

本轮明确不做以下事项：

- 不做多专题完整展开
- 不做外部互联网搜索
- 不做新信源补充
- 不做运维状态并入专题判断
- 不做长期记忆
- 不做专题 dossier 系统
- 不反向影响抓取、分类、渲染主链路

第三版仍然是：

**一个站在日报、当日情报判断和跨日观察之上的最小专题跟踪 agent。**

## 5. 推荐方案

采用 `候选池 + 主专题` 方案。

即：

- 系统先识别最近几天值得盯的 2-3 个候选主题
- 再从中选出 1 个主专题
- 只对主专题生成完整的专题跟踪结果
- 其他候选主题只保留在候选列表中，不在页面展开成长内容

不采用：

- 每天只盯 1 个主题、完全不保留候选池的方案
- 每天完整展开 2 个或更多专题的方案

原因：

- 只盯单一主题会让选择依据不透明
- 多专题完整展开会把页面和 Markdown 很快重新拉长
- `候选池 + 主专题` 兼顾了广度与收敛度，更符合当前产品节奏

## 6. Agent 职责边界

第三版专题跟踪 agent 只做五件事：

1. 读取最近 N 天日报
2. 读取最近 N 天 `daily_intel_brief`
3. 读取最近 N 天 `cross_day_intel_brief`
4. 识别候选主题并选出一个主专题
5. 输出两种消费形式：
- `theme-tracking-brief.md`
- 页面中的 `专题跟踪` 区块

它应回答的问题包括：

- 最近几天最值得继续盯的专题是什么
- 这个专题为什么今天值得继续看
- 哪些公司在持续参与这个专题
- 各家公司在这个专题中的切入点差异是什么
- 这个专题最近几天的演化是升温、扩展、分化还是回落
- 明天是否还值得继续跟踪

## 7. 输入设计

第三版只消费现有产物，不直接接触抓取器与原始信源。

### 7.1 日报输入

路径：

- `build/site/<date>/report.json`

用途：

- 获取主题聚合、代表条目、参与公司、热点主题等基础内容信息

### 7.2 v1 输入

路径：

- `build/site/<date>/daily_intel_brief.json`

用途：

- 复用已有的当日内容判断
- 避免专题跟踪重新解释一遍“今天最值得关注什么”

### 7.3 v2 输入

路径：

- `build/site/<date>/cross_day_intel_brief.json`

用途：

- 复用已有的跨日判断
- 直接利用 `warming_themes`、`steady_companies` 等跨日结果
- 避免专题跟踪自己重做一遍跨日分析

### 7.4 输入原则

- 不直接读取抓取器中间状态
- 不重新抓 HTML / RSS
- 不让专题跟踪层重新构造事实
- 只消费已经落盘的日报、v1 和 v2 结果

## 8. 统一分析结果设计

建议新增统一对象：

- `theme_tracking_brief.json`

推荐路径：

- `build/site/<date>/theme_tracking_brief.json`

结构至少包含：

- `date_range`
- `candidate_themes`
- `primary_theme`
- `theme_summary`
- `participating_companies`
- `company_angles`
- `theme_evolution`
- `continue_tracking`
- `next_day_theme_focus`
- `mode_used`

### 字段说明

#### `candidate_themes`

最近几天值得盯的 2-3 个候选主题。

#### `primary_theme`

最终选中的主专题。

#### `theme_summary`

这个专题为什么值得今天继续看。

#### `participating_companies`

持续参与该专题的公司列表。

#### `company_angles`

公司与切入点的映射，例如：

- `OpenAI`: 平台与安全机制前置
- `Google`: 产品功能落地
- `Microsoft`: 企业落地与客户场景

#### `theme_evolution`

该专题最近几天是升温、扩展、分化还是回落。

#### `continue_tracking`

是否建议明天继续跟踪该专题。

#### `next_day_theme_focus`

如果继续跟踪，明天最应该看什么。

#### `mode_used`

标记这份专题跟踪结果是 `rule / llm / hybrid`。

## 9. 在必要层使用 Agent / LLM 的原则

这轮需要明确区分“用 LLM 做表达”和“让 LLM 重做事实判断”的边界。

### 9.1 应当使用 LLM 的层

专题跟踪里的这些结果适合使用现有 LLM 能力增强表达：

- `theme_summary`
- `company_angles`
- `theme_evolution`
- `next_day_theme_focus`

原因：

- 这些字段更偏分析表达
- 更适合利用现有 capability layer 的 `topic_comparison` 和 `daily_editorial` 风格
- 能显著提升“像分析博客”的阅读感

### 9.2 不应交给 LLM 的层

下面这些部分不应让 LLM 自行决定：

- 候选主题的原始来源
- 参与公司的客观集合
- 主题是否在最近几天出现
- 主专题候选池的事实边界

原因：

- 这些属于结构化事实判断
- 更适合规则层与现有产物驱动
- 可以减少幻想和越界分析

### 9.3 推荐策略

推荐采用：

- `规则层选候选`
- `规则层确定边界`
- `LLM 层生成分析表达`

也就是说，专题跟踪 agent 本身是一个更高层的 agent 能力，但在内部仍遵循当前项目已经验证过的 `hybrid` 路线。

## 10. 输出设计

### 10.1 Markdown 报告

路径建议：

- `build/site/<date>/theme-tracking-brief.md`

结构建议：

- 标题
- 主专题
- 为什么今天值得继续看
- 参与公司与差异
- 主题演化
- 是否继续跟踪
- 明日关注点

这是第三版专题跟踪 agent 的完整文本产物，适合归档与人工 review。

### 10.2 页面区块

建议在当日日报详情页新增：

- `专题跟踪`

页面中只放高价值短块：

- 主专题
- 主题摘要
- 参与公司
- 差异点
- 是否继续跟踪
- Markdown 报告入口

页面区块不应成长文，而应保持“快速提取 + 深读链接”的节奏。

## 11. 模块划分

建议新增四个模块。

### 11.1 `theme_tracking_input.py`

职责：

- 读取最近 N 天的：
  - `report.json`
  - `daily_intel_brief.json`
  - `cross_day_intel_brief.json`
- 转成统一的专题跟踪输入

### 11.2 `theme_tracking_analysis.py`

职责：

- 选出候选主题
- 选出主专题
- 提取参与公司
- 提取公司切入点
- 生成主题演化与继续跟踪判断

这是第三版 agent 的核心逻辑层。

### 11.3 `theme_tracking_outputs.py`

职责：

- 输出：
  - `theme_tracking_brief.json`
  - `theme-tracking-brief.md`
  - 页面区块可消费数据

### 11.4 `theme_tracking_pipeline.py`

职责：

- 编排整个专题跟踪链：
  - 读取输入
  - 执行分析
  - 写出结果
  - 把专题区块挂到当日日报渲染输入

## 12. 接入方式

推荐顺序：

1. `pipeline.py` 生成主日报
2. `agent_pipeline.py` 生成 v1 当日情报判断
3. `cross_day_pipeline.py` 生成 v2 跨日观察
4. `theme_tracking_pipeline.py` 生成 v3 专题跟踪
5. `render.py` 若检测到专题产物，则渲染 `专题跟踪` 区块

这样做的好处：

- 主日报仍然是最稳定的主链
- v1、v2、v3 是逐层增强
- 任一增强层失败时都可以安全降级
- 后续若扩展到“多专题”或“专题 dossier”，也可以继续挂在这条链后面

## 13. 降级与兼容策略

### 13.1 主链路兼容

- `generate`
- `generate-today`
- `backfill`
- 自动化脚本

这些入口都不应因为 v3 失败而中断主日报生成。

### 13.2 输入降级

- 缺少部分 `daily_intel_brief.json` 时，继续使用 `report.json`
- 缺少部分 `cross_day_intel_brief.json` 时，继续保守生成候选主题
- 历史窗口不足时，输出更保守的专题判断，而不是强行写“演化趋势”

### 13.3 页面降级

- 若 v3 未生成成功，则页面不渲染 `专题跟踪`
- 保留已有的 `情报判断` 与 `跨日观察`

## 14. 测试策略

本轮建议至少覆盖：

1. `theme_tracking_input`
- 正常读取最近 N 天输入
- 缺少部分 v1/v2 输入时仍能继续

2. `theme_tracking_analysis`
- 正确识别候选主题
- 正确选出主专题
- 正确提取参与公司
- 正确生成是否继续跟踪判断

3. `theme_tracking_outputs`
- 正确生成 JSON
- 正确生成 Markdown
- 页面区块结构稳定

4. `theme_tracking_pipeline`
- 成功写出产物
- 失败时不破坏主日报

5. `render`
- 有专题产物时出现 `专题跟踪`
- 无专题产物时页面仍正常

## 15. 风险与控制

### 风险 1：候选主题选择过于随意

控制：

- 先由规则层给出候选主题池
- 再由分析层在候选池内做主专题选择

### 风险 2：主题演化判断过度外推

控制：

- 历史窗口不足时保守输出
- 不从 1-2 条偶发内容里写过重结论

### 风险 3：页面重新变得太长

控制：

- 页面只展示一个主专题
- 长文本继续放在 Markdown 报告中

### 风险 4：专题判断与跨日观察重复

控制：

- 让 v2 负责“最近几天哪些变化在发生”
- 让 v3 负责“哪一个主题最值得继续跟踪，以及为什么”

## 16. 完成标准

这轮完成的判断标准是：

1. 能从最近 N 天日报中选出候选主题与主专题
2. 能生成 `theme_tracking_brief.json`
3. 能生成 `theme-tracking-brief.md`
4. 当日日报详情页能展示 `专题跟踪` 区块
5. v3 失败时主日报、v1 与 v2 仍然正常生成
6. 测试覆盖候选选择、主专题选择、降级路径与页面接入

达到以上条件后，可以认为第三版专题跟踪 agent 的基础闭环已经建立。
