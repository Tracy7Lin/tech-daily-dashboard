# 前端对话 Agent 设计文档

## 1. 目标

为现有科技日报增加一个可直接在页面中提问的前端对话入口。

这不是一个通用聊天框，也不是一个会自主搜索互联网的重型 agent。第一版目标更明确：

- 让用户可以基于当前日报、跨日观察、专题跟踪和运维快照直接提问
- 提供比静态卡片更灵活的访问方式
- 保持现有日报页面的科技感、简约风格与快速提取能力

第一版对话 agent 应回答的问题包括：

- 今天最值得关注的是什么
- 某家公司最近几天在做什么
- 为什么今天的主专题是某个主题
- 最近哪些主题在升温
- 现在哪些信源仍有问题

## 2. 非目标

第一版不做以下能力：

- 自主抓取互联网或补充外部信息
- 改写日报主链路
- 长期记忆或用户画像
- 多轮工具编排
- 真正的运维修复能力
- 开放式闲聊

也就是说，它是一个：

`消费现有日报产物并进行受控问答的页面内对话 agent`

## 3. 推荐方案

采用：

`页面右下角浮动聊天入口 + 基于本地产物回答的对话 agent v1`

不采用全页聊天布局，也不采用单独新页面。原因如下：

- 不破坏现有日报阅读流
- 适合“先看日报，再问细节”的使用方式
- 与现有现代产品感页面风格更一致
- 可以把对话视为增强层，而不是新的主入口

## 4. 前端形态

### 4.1 入口

在页面右下角增加一个固定浮动入口：

- 文案建议：`情报问答`
- 视觉风格：
  - 圆角胶囊或圆形按钮
  - 延续当前蓝青科技感配色
  - 有轻量阴影与悬浮反馈

### 4.2 面板

点击后打开右侧抽屉式对话面板，而不是全屏 modal。

原因：

- 已有内容详情使用 modal
- 对话更适合持续停留与上下文滚动
- 抽屉式布局更像成熟 SaaS/AI 产品体验

建议结构：

- 顶部：
  - 标题：`情报问答`
  - 副标题：`基于今日日报、跨日观察与专题跟踪回答`
  - 关闭按钮
- 中部：
  - 对话消息流
- 底部：
  - 输入框
  - 发送按钮
  - 可选快捷问题 chips

### 4.3 风格要求

延续现有产品视觉语言：

- 简约
- 科技感
- 清晰层级
- 不做花哨渐变堆叠
- 保持高信息密度但不压迫

建议具体方向：

- 背景仍使用浅色科技感渐变体系
- 面板卡片采用磨砂白底 + 细边框
- 用户消息与 agent 消息有清晰但克制的区分
- 快捷问题 chips 采用低饱和高亮色块
- 强调“可扫读”，而不是聊天软件感

## 5. 数据输入边界

第一版对话 agent 只允许读取现有产物：

- `build/site/<date>/report.json`
- `build/site/<date>/daily_intel_brief.json`
- `build/site/<date>/cross_day_intel_brief.json`
- `build/site/<date>/theme_tracking_brief.json`
- `build/data/health_snapshot.json`

可选地读取最近几天同类产物，用于回答跨日问题。

明确禁止：

- 直接读取抓取器原始返回
- 直接发起外网搜索
- 自主访问 RSS / HTML 官方源

这样可以保证：

- 事实边界稳定
- 问答内容与日报一致
- 不会因为聊天引入新的事实漂移

## 6. 能力边界

第一版建议新增一个对话能力层：

- `chat_agent_input`
- `chat_agent_analysis`
- `chat_agent_response`
- `chat_agent_pipeline`

职责建议如下：

### 6.1 `chat_agent_input`

职责：

- 加载当日日报及相关产物
- 加载必要的最近几天摘要
- 将页面问题转成统一上下文输入

### 6.2 `chat_agent_analysis`

职责：

- 对用户问题做意图归类
- 判断回答应主要基于：
  - 当日情报判断
  - 跨日观察
  - 专题跟踪
  - 运维状态
- 提取最相关上下文

### 6.3 `chat_agent_response`

职责：

- 在必要层调用现有 LLM 能力生成回答
- 保持回答简洁、可信、边界明确
- 不允许扩写到当前产物之外

### 6.4 `chat_agent_pipeline`

职责：

- 串起输入、分析、回答生成
- 输出前端可直接消费的回答对象

## 7. 问题分类

第一版建议先支持 4 类问题：

1. `daily_summary`
- 例如：今天最值得关注什么

2. `company_focus`
- 例如：OpenAI 最近几天在做什么

3. `theme_focus`
- 例如：为什么今天的主专题是安全与治理

4. `ops_status`
- 例如：现在还有哪些信源有问题

不在这四类内的问题，先做温和降级：

- 明确说明当前问答主要基于日报产物
- 引导用户改问更贴近日报内容的问题

## 8. 输出格式

前端对话回答建议采用结构化响应，再渲染成消息。

建议字段：

- `answer`
- `question_type`
- `sources_used`
- `follow_up_suggestions`
- `mode_used`

其中：

- `answer`：主回答
- `question_type`：问题类别
- `sources_used`：用了哪些产物
- `follow_up_suggestions`：推荐继续问什么
- `mode_used`：`rule / llm / hybrid`

## 9. 回答风格

回答风格应接近：

`日报编辑 + 情报助理`

要求：

- 简洁
- 结构清晰
- 不假装知道当前产物之外的信息
- 不使用空泛套话
- 尽量给出基于日报产物的明确判断

建议风格：

- 先给结论
- 再给 1-3 个依据
- 最后给可继续追问的问题

## 10. 接入方式

第一版不引入独立后端服务。

推荐实现方式：

- 在本地生成阶段额外生成一个可供前端问答消费的上下文快照
- 页面侧通过轻量交互调用本地问答逻辑

如果需要最小交互入口，可以通过现有 Python CLI 增加一个本地问答命令，再由前端桥接调用。

更稳的工程实现顺序建议是：

1. 先把问答逻辑做成独立 Python 模块
2. 再补一个最小 CLI 问答入口
3. 最后再接页面右侧抽屉 UI

## 11. 失败与降级策略

第一版必须允许问答能力失败而不影响日报页面本身。

要求：

- 对话 agent 初始化失败时，页面仍能正常打开
- LLM 不可用时，应回退到规则或模板回答
- 无法回答的问题应给出边界说明，而不是编造答案

## 12. 实现优先顺序

推荐顺序：

1. 问答设计文档
2. 问答能力层
3. 最小 CLI 入口
4. 页面抽屉 UI
5. 快捷问题与前端体验优化

## 13. 验收标准

完成第一版后，应满足：

- 页面出现稳定、风格一致的对话入口
- 可以回答四类核心问题
- 回答内容不超出当前日报产物边界
- LLM 失败时可降级
- 不破坏现有日报主链路
- 前端风格与现有看板统一
