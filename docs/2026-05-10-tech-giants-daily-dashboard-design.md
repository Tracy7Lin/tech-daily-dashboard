# 科技巨头每日情报看板 v1 设计文档

## 1. 设计目标

本设计文档用于固化 `科技巨头每日情报看板 v1` 的产品结构与信息组织方式，为后续实现提供稳定边界。

v1 的核心原则如下：

- 主题优先，而不是公司列表优先
- 官方优先，而不是全网聚合优先
- 高准确率优先，而不是高覆盖率优先
- 速览优先，而不是长文分析优先
- 静态站点落地，但内部结构保留后续扩展空间

## 2. 整体产品结构

站点采用两层结构：

- 首页：当天总览、主题对比、公司动态摘要
- 每日日报详情页：当天完整内容与完整原文入口

同时提供历史归档页，支持按日期浏览过往日报。

## 3. 首页信息架构

### 3.1 今日总览

首页顶部提供当天的全局判断，目标是让用户在数秒内知道“今天发生了什么”。

建议包含以下信息：

- 日期
- 今日有效新闻数
- 覆盖公司数
- 今日最热主题 1-3 个
- 一句话趋势判断

趋势判断应是压缩后的编辑式结论，例如：

> 今天的核心信号是，AI agent 正从模型能力竞争转向企业落地与终端入口竞争。

### 3.2 主题对比

首页主区块应以 `主题对比` 为核心，而不是以公司列表为核心。

每个主题卡片聚焦一个方向，例如：

- AI agent
- 新模型发布
- 企业 AI 平台
- 开发者生态
- 终端 AI 入口

每个主题卡片应包含：

- 主题名
- 主题摘要
- 相关公司
- 差异对比
- 趋势判断
- 关联原文入口

其中 `差异对比` 应以并列、简洁的判断句输出，而不是长篇说明。

### 3.3 公司动态

首页下半部分展示公司维度的当天动态，用于补足主题视角下未完全展开的信息。

每条公司动态应包含：

- 标题
- 中文摘要
- 分类标签
- 发布时间
- 原文链接
- 重要度

同一家公司当天有多条内容时，排序规则为：

1. 先按重要度排序
2. 再按发布时间排序

## 4. 详情页结构

每日日报详情页负责承载当天完整内容，建议保留以下区块：

- 当日总览
- 全部主题对比
- 全部公司动态
- 源状态说明

其中 `源状态说明` 用于明确哪些公司当天无有效动态、哪些信源抓取失败，以保证结果透明可信。

## 5. 历史归档结构

首版历史归档按日期组织，不引入复杂检索。

每个日期入口建议展示：

- 当天标题摘要
- 当天主题数
- 当天重点公司
- 进入日报详情页的链接

后续可在不破坏现有结构的前提下扩展：

- 按公司筛选
- 按主题筛选
- 按标签浏览

## 6. 内容处理规则

### 6.1 采集规则

只采集满足以下条件的内容：

- 发布时间落在当日日报周期内
- 来源属于官方博客、官方公告、官方开发者博客、官方更新页或官方社媒补充入口
- 内容命中高价值类别：AI / 产品发布 / 平台能力 / 生态合作 / 组织与战略 / 未来方向
- 对需要依赖文章页元信息补全发布日期的 HTML 索引源，若无法提取可信发布时间，则该条内容不进入当日日报

对明显低价值内容执行降权或过滤。

### 6.2 摘要规则

每条新闻统一生成中文速览摘要，默认控制在 1-3 句，并回答：

- 发生了什么
- 为什么重要
- 它更像产品动作、技术动作还是战略动作

对极重要内容可补一句影响判断，但不进入长文模式。

### 6.3 对比规则

系统先基于内容打主题标签，再按相似主题进行自动聚合。

对比输出需要回答：

- 哪几家公司在做同一类事
- 它们的切入点分别是什么
- 谁偏产品落地，谁偏基础能力，谁偏生态布局
- 这说明了什么趋势

## 7. 运行与可信度要求

日报采用全自动生成方式，按北京时间自然日运行。

可信度要求如下：

- 宁可少报，也不引入大量噪音
- 所有摘要都必须能追溯到官方原文
- 页面需要如实呈现“今日无有效动态”与“信源暂不可用”等状态

## 8. 设计约束

v1 不包含以下内容：

- 账号系统
- 团队协作能力
- 评论与收藏
- 全网非官方媒体聚合
- 长篇研究报告式分析

## 9. 后续扩展方向

虽然 v1 采用静态站点形态，但设计上保留以下升级空间：

- 观察名单改为配置化
- 信源配置扩展
- 主题体系持续调整
- 首页增加按主题或公司切换视图
- 站点升级为可检索、可筛选的动态看板
