# 科技日报项目改进路线图

## 1. 文档目的

这份文档用于收束当前 `tech-daily-dashboard` 项目的后续优化方向，避免后续工作继续以零散问题驱动推进。

目标不是一次性规划所有未来能力，而是给出一份可执行、可验收、可迭代的改进路线图，供后续开发按批次落地。

## 2. 当前项目状态

截至当前版本，项目已经具备以下能力：

- 稳定的日报主链路：抓取、过滤、分类、摘要、主题聚合、静态渲染
- 混合表达层：规则 + LLM，且具备 fallback
- 本地自动化：`generate-today`、PowerShell 脚本、Windows 定时任务注册
- 基础运维可观测性：`health-check`、`dry-run`、源级配置诊断、最近一次运行诊断、历史运行诊断、高优先级问题摘要、最近恢复问题摘要、结构化 `health_snapshot`
- 前端摘要化展示：卡片 + modal 查看详情 + 页面内情报问答抽屉
- 对持续不可抓取或低产出的公司提供占位说明，而不是静默空白
- 当日日报、跨日观察、专题跟踪、前端问答四层 agent 增强链已经打通

当前项目已经超出“原型脚本”的阶段，属于：

`可持续迭代的个人情报日报产品雏形`

## 3. 已完成的改进线

以下工作已经不再是待办，而是当前基线能力：

### 3.1 自动化与运维基线

- `generate-today`
- `health-check`
- `dry-run`
- `run_daily_report.ps1`
- Windows 定时任务注册/删除脚本
- 运行日志写入 `build/logs`
- 生成后自动追加一次健康检查

### 3.2 内容表达基线

- `rule / llm / hybrid` 三种模式
- DeepSeek 兼容接入
- 单条摘要、首页判断、主题对比的 LLM 化表达层
- LLM 输出后处理与 fallback
- agent-ready capability layer 基线：
  - `brief_generation`
  - `topic_comparison`
  - `daily_editorial`
  - `ops_status_analysis`
- 最小情报分析 agent 基线：
  - `daily_intel_brief.json`
  - `agent-brief.md`
  - 日报详情页中的 `情报判断` 区块
- 跨日追踪 agent 基线：
  - `cross_day_intel_brief.json`
  - `cross-day-brief.md`
  - 日报详情页中的 `跨日观察` 区块
- 专题跟踪 agent 基线：
  - `theme_tracking_brief.json`
  - `theme-tracking-brief.md`
  - 日报详情页中的 `专题跟踪` 区块

### 3.3 前端基线

- 现代产品感的页面风格
- 摘要卡片 + modal 展开
- 公司空状态与信源不稳定状态占位
- 前端对话 agent v1：
  - CLI `chat`
  - 页面右下角浮动入口
  - 右侧抽屉式聊天面板
  - `rule / llm / hybrid` 问答路径
  - 加载中 / 回退 / 降级状态
  - 焦点管理与基础移动端适配

### 3.4 源级诊断基线

- 最近一次运行问题
- 最近几天重复问题
- 高优先级问题摘要
- 已恢复问题不再继续出现在高优先级列表
- 区分“信源稳定但当日无动态”和“信源暂未稳定”

### 3.5 Agent 增强链基线

- v1：当日情报判断
- v2：跨日观察
- v3：专题跟踪
- v4 前置能力：页面内对话问答入口

## 3.6 当前真正未完成的主线

当前不再缺少日报主链路、运维闭环或前端问答基线，真正还未正式启动的下一阶段是：

- `Theme Dossier Agent`

它将位于现有 `专题跟踪 agent` 之后，目标是把“今天最值得盯的主题”进一步沉淀成一个可持续复用的小型专题档案，而不是再添加一层单日摘要。

## 4. 当前仍值得推进的改进方向

剩余改进方向分为 5 类。

### 4.1 内容质量

目标：

- 提升日报可读性和长期阅读价值

当前问题：

- 单条摘要质量仍有波动
- 主题 summary / comparison / trend 的文风还可以更统一
- `其他重要动态` 仍可继续压缩
- LLM 生成内容仍有少量“分析力度不足”或“判断略平”的情况

说明：

这条线价值高，但当前不作为最近一批优先事项。

### 4.2 信源可靠性

目标：

- 提高真实抓取成功率和稳定产出能力

当前问题：

- `Tesla` 官方源持续 `403`
- `Xiaomi` 官方 Discover 页主要依赖动态渲染，当前静态抓取器无法稳定拿到文章链接
- 仍有部分公司需要长期观察，防止源结构变化后重新退化

说明：

这条线仍然重要，但当前应按“低风险增量修复”推进，不适合为单一公司引入高复杂度抓取方案。

### 4.3 运维与可观测性

目标：

- 让系统更像可维护的长期运行产品，而不是一次性任务

当前问题：

- 当前主链已基本补齐
- 后续若继续推进，重点不再是补底层运维能力，而是消费这些运维产物

说明：

这条线已经达到可持续运行的基线，短期内不再是最高优先级。

### 4.4 前端体验收尾

目标：

- 提升长期阅读舒适度和交互完成度

当前问题：

- modal 可访问性仍可继续提高
- 结论区 / 证据区的层次仍可更清晰
- 对话抽屉仍可继续做更强的多轮与证据化体验

说明：

这条线当前属于可选精修，而不是主线缺口。

### 4.5 兄弟日报与未来扩展

目标：

- 在主日报成熟后，拓展新的 sibling pipeline

当前状态：

- `GitHub 今日 Highlight 日报` 已有概念文档
- 还未进入正式需求设计与实现

说明：

这条线属于中期扩展，不应抢占主日报当前的稳定性改进资源。

### 4.6 主题研究深化

目标：

- 把现有 `专题跟踪` 从“单日重点主题追踪”升级成“可持续复用的专题档案”

当前问题：

- 目前只有 `theme_tracking_brief.json` 和 `theme-tracking-brief.md`
- 还没有：
  - 主题状态机
  - 主题时间线
  - 可持续积累的 dossier 结构
  - 页面中的 dossier 轻入口

说明：

这条线是当前最值得启动的下一阶段。

## 5. 推荐优先级

基于当前项目状态，推荐优先级如下：

1. `运维与可观测性`
2. `主题研究深化`
3. `信源可靠性`
4. `内容质量`
5. `前端体验收尾`
6. `兄弟日报与未来扩展`

## 6. 下一阶段推荐批次

后续优化按批次推进，每批次都应满足：

- 有明确目标
- 有清晰验收口径
- 尽量低耦合
- 改动后必须回归测试

### 批次 A：运维闭环补全

目标：

- 让 `health-check` 从“发现问题”进一步进化为“跟踪问题生命周期”

建议任务：

- 增加 `recently_recovered_runtime` 摘要
- 在日志中突出显示“恢复中的公司”
- 为高优先级问题加入更紧凑的摘要输出
- 视情况增加 `health-check --verbose` 风格扩展

验收口径：

- 能同时看见当前问题、高优先级问题、最近恢复问题
- 自动化日志能够体现“出问题”和“恢复”两个方向

状态：

- 已完成，不再作为当前待执行批次

### 批次 A2：Theme Dossier Agent 启动

目标：

- 在现有 `专题跟踪 agent` 之上，引入可持续复用的主题 dossier 结构

建议任务：

- 设计 `theme_dossier.json`
- 增加 `theme-dossier.md`
- 增加主专题状态机：`emerging / active / fragmenting / cooling`
- 增加 3-6 条代表事件的专题时间线
- 页面详情页只增加轻量 dossier 入口，不把日报做重

验收口径：

- 主专题不再只是一段短摘要，而是形成可持续查看的档案
- 页面仍然保持轻量，Markdown 和 JSON 承担完整 dossier 细节
- 后续 agent 可直接消费 dossier 作为长期专题上下文

### 批次 B：信源持续巡检

目标：

- 避免历史修好的公司再次退化

建议任务：

- 对已修好的 `ByteDance / Alibaba / Huawei / Tencent / xAI` 做持续观察
- 增加简单的“连续 3 天低产出提醒”
- 明确 `Tesla / Xiaomi` 维持占位策略，除非出现低复杂度可行官方替代源

验收口径：

- 已修好的公司不会因旧历史噪音继续占据高优先级
- 低产出风险能被更早发现

状态：

- 进行中，但短期保持低复杂度增量维护

### 批次 C：内容质量第二轮提升

目标：

- 把日报从“可读原型”继续推向“高质量博客式日报”

建议任务：

- 统一 `summary / comparison / trend` 文风
- 压缩 `其他重要动态`
- 优化 LLM 提示词与输出后处理
- 强化“为什么重要”的表达稳定性

验收口径：

- 同一天不同卡片之间文风更统一
- 低信息密度主题减少
- 用户主观阅读体验明显改善

状态：

- 已明确 postponed，等待主题研究深化后再决定是否优先恢复

### 批次 D：前端收尾

目标：

- 提高长期阅读效率与交互完成度

建议任务：

- modal 焦点管理
- modal 中结论区 / 证据区分层
- 移动端微调
- 主题卡主次增强

验收口径：

- modal 阅读更顺手
- 键盘关闭与焦点回退行为更完整
- 移动端不再出现阅读节奏明显不稳的问题

状态：

- 基线已完成，剩余为可选精修

### 批次 E：兄弟日报启动

目标：

- 在主日报稳定的前提下，启动新的 sibling pipeline

建议任务：

- 将 `GitHub 今日 Highlight 日报` 从概念文档推进到需求文档
- 定义热度信号、入选标准、主题体系、页面结构

验收口径：

- 输出一套独立需求与设计文档
- 不与现有科技日报主链路耦合

## 7. 当前推荐立即执行项

如果后续继续按这份文档推进，我建议下一步直接进入：

`批次 A：运维闭环补全`

也就是：

1. 增加 `recently_recovered_runtime` 摘要
2. 让自动化日志同时体现“当前高优先级问题”和“最近恢复问题”
3. 继续把 `health-check` 打造成真正可日常依赖的运维入口

## 8. 执行原则

后续每一批改进都遵循下面的原则：

- 先补测试，再改实现
- 优先保持高内聚、低耦合
- 优先在现有边界内演进，而不是扩散新层
- 不因为个别边缘问题引入高复杂度方案
- 每次改动都要能独立回滚、独立验证、独立理解

这份文档后续应作为项目改进的统一参考入口。
